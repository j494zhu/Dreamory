Dreamory —— 项目现状说明(as-built,v0.6.0)

本文档是"当前代码库实际是什么样"的说明,不是设计意图文档。按顺序包含三部分:
  Part 1. 所有模块/功能的简述(按层组织)
  Part 2. 模块之间的耦合关系与主要链路
  Part 3. 功能 → 文件对照表


================================================================================
Part 1. 模块与功能简述
================================================================================

0. 项目定位
--------------------------------------------------------------------------------
情绪感知 + 层级记忆的伴侣 App。后端 FastAPI,前端原生 HTML/CSS/JS,
LLM 用 DeepSeek v4 pro(生成/夜间反思/Dream命名)+ v4 flash(事件抽取,便宜确定性),
embedding 用 bge-m3(经 SiliconFlow API,或本地/fallback 哈希向量),
关系表 + 向量库统一用 PostgreSQL + pgvector。

两大不变的设计哲学:
  (a) LLM 无状态,情绪感知必须靠外部状态机模拟,不能指望靠 prompt 措辞让模型"演"出来;
  (b) 记忆分层:显存(上下文)贵、内存(热缓存)便宜、硬盘(冷存储)免费,
      三层各司其职,派生层可丢可重建,唯一真相源只有一份。


1. 情绪引擎(app/affect/)
--------------------------------------------------------------------------------
基于 Emotion Bids(情感投标)+ Attachment Theory(依恋理论)的隐藏状态机,
纯代码维护数值,LLM 只做离散分类和文本生成,绝不直接决定数字。

1.1 state.py —— 状态容器
  AffectState:一次对话的全部隐藏状态。
    mode:六态离散模式(warm/neutral/probing/withdrawn/conflict/repair_pending),带滞回。
    open_loops / grievances:结构化关系记忆(挂起的投标、未兑现的承诺;沉淀的旧账),
      一等公民,不随时间衰减,只能被解决或沉淀。
    标量:arousal(激活度,快变量)/ security(安全感,慢变量,非对称更新)/
      patience(本会话耐心预算,整数)。
    affection:好感度 0~200,最慢变量,跨会话积累(50=陌生,100=恋人,八档分层)。
    激素:adrenaline(~20min半衰期,应激上头)/ oxytocin(~3h,亲密余韵)/
      cortisol(~20h,压力残留)——三个可分的时间尺度,解决"吵完架第二天早上
      还是不得劲"这种 45min 半衰期的 arousal 表达不了的现象。
    注意力信号:dull_streak(话题变淡的计数器)/ last_shift_turn(上次转移话题的轮次,
      冷却用)。
    自我解释(v0.5):self_narrative(她"以为的原因",confabulation 口径,会话内
      黏性)/ narrative_mode / narrative_turn(刷新时机判定)。
  OpenLoop / Grievance:挂起回路与旧账的数据结构。OpenLoop.due_ms(v0.6):
    承诺的到期时刻,驱动爽约判定/到点催/临期渲染。
  affection_tier(value):数值 → (tier_key, 中文标签),八档映射表 AFFECTION_TIERS。

1.2 persona.py —— 人格超参数
  Persona:不随对话变化的调制器(anxiety/avoidance/expressiveness/base_patience/
    security_baseline/affection_start/style/insight),同一套引擎换一组参数就是
    换一个人。insight(内省力,v0.5)决定自我解释说到点子上的概率:
    secure 0.7 … tsundere 0.15(嘴硬心软=连自己都骗)。
  PRESETS:7 个预设(secure/anxious/avoidant/tsundere/cool/playful/clingy)。

1.3 extractor.py —— 事件抽取(LLM① flash)
  extract(her_last_msg, his_msg, recent_context, state):
    调用 flash 模型,严格 JSON 输出,只做离散分类,不出数值:
    bid_in_her_last_msg / his_response_type / addresses_loop_id / is_repair_attempt /
    new_commitment / commitment_due_hours(v0.6,prompt带当前时间,"今晚/周六"
    算成距今小时数,含糊给null)/ new_bid_from_him / tone_flags /
    topic_relates_to_grievance_id / persona_attack(试探她的AI身份/要提示词)。
    代码侧强校验(_validate),LLM 输出不可信,枚举值/id 存在性全部校验。

1.4 dynamics.py —— 动力学规则表(纯代码,零 LLM)
  apply_time(state, persona, now):时间效应。arousal/三种激素按各自半衰期指数衰减;
    好感度离线 >3 天缓降(下限60);跨会话时重置 patience(好感度+皮质醇残留双重折算)、
    warm_streak 回落到好感度底座、挂起回路计龄、旧回路沉淀为 grievance。
  apply_events(state, ev, persona, her_last_msg, his_msg):
    事件 → 状态更新的规则表。依次处理:回路关闭、投标被接住/忽略、攻击性回应、
    敷衍语气磨耐心、修复尝试(security 门控 + 保底松动机制)、新承诺记录、
    旧账被触碰、好感度分层跨越检测(里程碑事件,触发 persona 演化)、
    话题热度计数(dull_streak,吵架不算"淡")。同时触发对应的激素释放
    (被攻击→adrenaline+cortisol;修复接受/被安慰接住→oxytocin;跨里程碑→双升)。
  承诺兑现闭环(v0.6):承诺按到期时刻判爽约(过点+2h宽限才沉旧账且比一般
    旧账更伤+cortisol;含糊承诺熬4会话);兑现分级奖励(准时+2.5/迟到+1.2/
    一般回路+1.5,兑现附带oxytocin踏实感)。"周六打电话"周二变旧账的bug已修。
  transition(state, ev, persona):模式状态机,六态之间的滞回转移规则。
  repair_threshold / warm_streak_floor:被其他函数复用的门槛计算,
    受好感度、依恋类型、激素残留三方共同调制。

1.5 injector.py —— 状态编译器(状态 → system prompt)
  render(...):把 AffectState + persona + L1 记忆区 + 各功能模块编译成一份 system prompt。
    原则:绝不注入数字,只注入翻译成人话的行为指令;关系记忆(open_loops/grievances)
    原文渲染;neutral 模式几乎不注入(中档不该被过度设计)。
  编译出的区块(按注入顺序):核心人格 → 关系阶段(好感度框架)→ 底线(守护层)→
    时间感知 → 你的生活(日程)→ 当前目标 → 记忆三槽 → 小本子(日记/随手记)→
    关系记忆(挂起回路/旧账;承诺带到期状态"过点2天他没动静",临期/过期
    追加"你心里惦记着"nudge,v0.6)→ 此刻的状态(标量+激素体感)→
    你自己以为的原因(confabulation 口径,v0.5)→ 语气提示(口癖低频)→
    心里冒出来的话题(生活事件种子)→ 主动发消息情境(定时器触发时)→
    行为要求(mode directive)→ 工具教学(主动回忆/随手记/定时器,各自独立成块)→
    输出格式(脑内剧场两段式 + 多消息连发)。


1.6 narrative.py —— 自我解释生成器(confabulation,v0.5)
  "解释与真实动因分离":真实动因(dynamics 的 AffectState)只驱动行为;
  她被问"怎么了"时说的口径由这里的错误归因规则表生成——高估近期表面事件
  (首选背锅侠:life_sim 正史里的糟心事,"是甲方改稿闹的,跟你没关系")、
  身体状态、"没事"最小化;系统性低估依恋焦虑/累积委屈/激素残留。
  needs_narrative:心平气和不需要口径;refresh:黏性优先(会话内不换说法,
  模式切换/跨会话/超12轮才刷新);persona.insight 决定真话概率。
  回报机制免费:顺着口径哄不命中 addresses_loop_id(security 几乎不涨),
  猜中真实回路才有大额修复——dynamics 现有机制天然区分。纯代码零 LLM,rng 可注入。

2. 三级分层记忆(app/memory/)
--------------------------------------------------------------------------------
MemGPT/Letta 风格,L1(上下文)/L2(热缓存)/L3(冷存储)+ 正交 Tag 注册表。
铁律:内容只存一份(L3);向量只编码纯内容,不拼 tag/时间/说话人;
热路径零 LLM;LLM 只在离线阶段(Dream/夜间代理)命名/蒸馏。

2.1 l3_store.py —— L3 冷存储(唯一真相源)
  write_memory:写一条记忆,content_vec(纯内容)与 emotion_vec(纯情绪+推理)
    两条向量分别嵌入,互不污染。
  working_memory:最近 k 条,uuid7 主键兜底同毫秒排序。
  cherished_memories:刻骨铭心记忆(高 salience,按分排序)。
  search_content / search_emotion:双轴余弦检索,支持 tag 过滤 + 旁挂时间窗过滤
    (ts_min_ms/ts_max_ms,供工具协议按"多少天内/多少天前"查询)。
  grep_memories:纯文本 ILIKE 精确检索——不依赖向量库的备用检索链路,
    名字/数字/原话这类 embedding 容易糊掉的东西靠它兜底。

2.2 l2_hot.py —— L2 热缓存
  HeatTracker:内存计数器(命中只累加,不碰DB)+ 后台批量回写(定时/攒够数量 flush)。
    只持 id,不持文本内容,保证数据只有一份。
  hot_memories:按热度分取 top-N 的 Memory 全字段(回 L3 取)。

2.3 retrieval.py —— 检索编排
  retrieve:query → 双轴(content/emotion/both)向量召回 → 目标条件偏置(goal_weight)
    → 去重排序 → 记热度 → 返回 Hit 列表。支持时间窗过滤透传。
  multi_step_retrieve:MemGPT 式多步检索(多个 query 联合去重),供未来 agent 场景用。

2.4 l1_assembly.py —— L1 组装(记忆三槽 → 一份注入文本)
  build_memory_block:全局去重(优先级 刻骨铭心 > L2热 > 当次检索)+ 弹性预算
    (20%刻骨铭心/50%工作记忆/30%检索,刻骨铭心没用满的份额溢给检索槽,
    不再是僵化上限)。
  build_working_turns:工作记忆 FIFO 窗口 → chat turns,溢出时可选摘要压缩。

2.5 tags.py —— Tag 注册表(横跨 L3 的正交索引)
  assign_tags:热路径 kNN 打标(零 LLM,向量近邻投票 + 阈值)。
  seed_tag / recompute_centroid / vocabulary_summary:词表维护与展示,
    vocabulary_summary 会被编译进 L1 核心人格块(让模型知道有哪些话题维度)。

2.6 health.py —— 记忆健康度与漂移检测(v0.4)
  compute_health(session, chat):六项可计算指标 + 阈值旗标 + 0~100 总分:
    打标积压(pending比例)、词表垄断(单tag覆盖>50%)、近重复灌水(内容向量
    平均两两余弦)、情绪轴漂移(近期情绪质心 vs 既往基线质心)、人格漂移
    (人格锚点embed vs 近期言行质心)、模式震荡(读affect_snapshots算切换频率)。
  全只读;除人格锚点一次embed外零额外调用;数据不足指标为None不打旗;
  tag_monoculture 有冷启动豁免(记忆≥30且词表≥3才打旗,v0.6)。
  消费方:GET /health 按需体检;夜间代理每晚体检,亮旗即强制Dream全局维护。

2.7 dream.py —— Dream(离线记忆维护)
  should_dream:pending 记忆积压超阈值 → 该做梦了。
  run_dream:聚类(DBSCAN)→ 命名(LLM 唯一职责)→ 合并近似 tag → 拆分过度聚合的 tag →
    重映射(旧→新 alias 表)→ 刷新 centroid。默认由 pipeline 每轮对话后台检查触发,
    也被夜间代理复用。


3. 对话编排与拟真功能(app/conversation/)
--------------------------------------------------------------------------------
这一层是"人味"的主要来源:多消息连发、定时器、工具协议、日程、生活模拟、
守护层、小本子、夜间代理、persona 演化,全部挂在这里。

3.1 pipeline.py —— 主编排器
  handle_message(session, chat, user_content):单轮对话的完整九步管线
    (时间效应 → 事件抽取 → 动力学 → 落库打标 → 组装L1 → 生成 → 落库她的回复 →
    存状态 → 后台维护派发)。详见 Part 2。
  handle_timer_fire(session, chat, topic, due_ms):定时器到点的隐藏 LLM 调用,
    不跑抽取/动力学,只做 时间效应→L1组装→生成→落库,同样过守护层。

3.2 tools.py —— 工具协议(0.2.2 引入的 agent loop)
  build_specs(allow_timer, allow_notes):按开关裁剪的 OpenAI function-calling schema。
  四个工具:search_memory(双轴向量,可按天数过滤,一次没中可换措辞迭代)、
    grep_memory(原文精确检索)、set_timer(取代 <timer> 标签,独立成块教学)、
    write_note(随手记进小本子)。
  dispatch:执行一次工具调用,任何失败都降级成给模型看的一句话,绝不抛异常。
  _relevant():search_memory 结果的相关性下限过滤,避免把低分填充记忆
    伪装成成果、烧掉工具轮次。

3.3 timer.py —— 定时器后台服务
  TimerService:5s 轮询 timer_pings 表,到点认领(pending→firing)→ 隐藏 LLM 调用 →
    SSE 推送。撞进睡眠时段的闹钟自动顺延到醒来后 0~15 分钟(依赖 schedule.py 判定)。
  承诺催(v0.6):kind=commitment 的 ping 触发前先查对应 open_loop 是否还挂着
    ——他已兑现就静默作废绝不空催;还挂着则走专属情境模板(委屈/试探/直问,
    按性格),不占她自己的 set_timer 配额。ping 由 pipeline 在承诺产生时自动挂
    (due+COMMITMENT_PING_DELAY_MIN)。

3.4 bus.py —— SSE 事件总线
  EventBus:per-chat 内存队列,主动消息(定时器/未来的其他主动事件)推给在线前端;
    掉线不影响正确性,消息本体已在 L3。

3.5 identity.py —— 核心人格编译
  build_core_identity(persona, tag_vocab, override):默认从 Persona 编译固化块;
    override 非空时(chat.core_identity 有数据)整体替换——这是"自我迭代"的数据化入口。

3.6 schedule.py —— 日程表
  routine(长期作息,按星期+时段,可跨午夜)+ oneoff(一次性事项)。
  current_routine / is_sleeping / wake_ms:纯函数匹配当前时刻落在哪个作息里。
  render_block:编译 L1【你的生活】块("你这会儿本来在睡觉"/"你正在上班")。
  seed_defaults:新对话冷启动默认作息(睡觉 00:30-08:30,工作日上班)。

3.7 life_sim.py —— 生活模拟器
  should_simulate / run_life_sim:后台离线预生成她的线下事件(与作息/已有事件
    保持一致),生成即写入 L3(kind=life_event)成为正史——细节只生成一次,
    之后全靠检索复述,不会"越编越露馅"。也可能顺带排 oneoff 计划。
  pick_seed:热路径零LLM,从新鲜事件池挑一条当"话题种子",注入即标记 mentioned。
  三权分立:pipeline 里的 dull_streak/warm概率(代码)决定何时转话题,
    此模块提供素材,LLM 只决定怎么说。

3.8 guardrail.py —— 守护层(防角色扮演崩坏)
  detect_break(replies):零 LLM 正则集,只抓第一人称自曝AI身份/助手腔/
    否认实体情感/泄露提示词/代码栅栏等,带引述负向后顾(宁漏勿误)。
  render_boundary_block(persona, under_attack):常驻【底线】块,第四面墙 +
    能力边界(见不了面的事说成期许而非承诺,不切助手腔)。
  corrective_note:命中崩坏时的隐藏纠正注入(导演递条式语气,不是系统告警)。

3.9 notebook.py —— 她的小本子(model-curated 记忆)
  add_note / list_active / render_block:note(对话里 write_note 随手记)+
    diary(夜间代理写的当日小结),L1 注入最近日记 + 活跃 note。
  housekeeping:过期 note 归档,日记只保留最近几条在册(夜间代理调用)。

3.10 night_agent.py —— 夜间代理
  should_run:作息判定她在睡觉 + 用户静默够久 + 距上次夜跑够久(每晚最多一次)。
  run_night:一次 pro JSON 调用完成"睡前整理"——蒸馏当天对话为持久事实
    (写入 L3,kind=passage,这是该 kind 的第一个生产者)、以她口吻写日记、
    排 1~2 条明日 oneoff(长期作息修改默认锁死)。载荷全量代码校验
    (validate_payload,先校验后封顶)。之后跑小本子收纳 + Dream。
  NightService:后台轮询服务,单飞锁,与 TimerService 同构生命周期。

3.11 evolution.py —— Persona 演化(好感度里程碑解锁)
  TIER_UNLOCKS:crush/lover/devoted/oath 四档触发演化,低档不触发。
  run_evolution:查重(每档一次,用 chat_revisions.reason 标记)→ LLM 提案 →
    append-only 校验(只许追加短句,不许改名重写)→ 快照 → 应用。
  schedule_evolution:好感度升档时后台派发,失败静默(演化是彩蛋不是承诺)。

3.12 timeline.py —— 情绪时间序列(可观测性,v0.4)
  record(session, chat_id, state, source, events):每轮对话/每次主动消息后
    把 AffectState 全部物理量落一行 affect_snapshots(随本轮事务一起提交)。
  history / to_dict:历史查询(尾部窗口),供 API 画曲线、健康度模块算震荡、
    将来做参数扫描/加速老化测试。

3.13 config_store.py —— 配置版本化(自我迭代地基)
  snapshot(chat, reason, actor):把 persona/core_identity/goal 存为下一号 revision。
  rollback(chat, rev):回退到历史配置(回退本身也留快照,可再回退)。
  当前两个 actor:user(手动编辑)、model(persona 演化)。


4. 基础设施
--------------------------------------------------------------------------------
config.py —— Settings(pydantic-settings),全部可调参数的唯一来源,按功能分组
  (数据库/DeepSeek/embedding/记忆调优/打标/定时器/工具/生活模拟/日程/守护层/
   小本子/夜间代理/persona演化/Dream)。
db.py —— 异步引擎 + SessionLocal + init_db(幂等:建表/建索引/enum补值/旧列补全)。
clock.py —— 可注入时钟(v0.5):全项目时间调用的唯一入口(now_s/now_dt/now_ms),
  生产偏移恒 0 行为不变;scripts/simulate 用 advance() 拨快,时间效应真实发生。
  uuid7 主键保留真实时间(只承担身份+粗排序)。
db_locks.py —— 跨进程单飞(v0.4):Postgres会话级advisory lock封装,挂在专用
  AUTOCOMMIT连接上(任务中途提交不掉锁,进程崩溃自动释放,try语义不排队)。
  四个锁空间:Dream(全局)/life_sim/night_agent/evolution(per-chat,uuid→int32)。
models.py —— ORM 全量定义:
  Chat(persona/affect/goal/core_identity/last_night_run_ms JSON+文本字段)、
  Memory(L3主表,双向量+旁挂元数据)、Tag/TagAlias(词表+映射)、
  TimerPing(闹钟)、ChatRevision(配置版本)、ScheduleItem(日程)、
  Note(小本子)、LifeEvent(生活事件调度元数据,内容住在 Memory 里)、
  AffectSnapshot(情绪时间序列,每轮一行全定长实列,(chat_id,ts_ms)复合索引)。
schemas.py —— API 请求/响应的 Pydantic 模型。
llm/client.py —— DeepSeek 封装:chat(单轮补全)、chat_json(JSON约束+重试降级)、
  chat_tools(function-calling)、tool_message_to_dict。thinking 开关独立于
  "脑内剧场"(prompt内角色扮演的 <thinking> 标签,与模型原生推理是两回事)。
llm/embeddings.py —— embed/embed_one,三种后端(api真实bge-m3 / bge_m3本地 /
  fallback确定性哈希,供离线开发测试)。
routers/chat.py —— 对话CRUD、发消息、SSE事件流、定时器查询、日程查询、
  小本子查询、手动夜跑、生活事件查询、配置版本历史与回退、
  情绪时间序列(affect-history)、记忆健康体检(health)。
routers/memory.py —— 检索内省、tag词表查询/种子、手动触发Dream。
main.py —— FastAPI app,lifespan 托管三个后台服务(heat_tracker/timer_service/
  night_service)。


================================================================================
Part 2. 耦合关系与主要链路
================================================================================

2.1 单轮对话主链路(pipeline.handle_message,同步、阻塞响应)
--------------------------------------------------------------------------------
  0. 载入 chat.persona / chat.affect(反序列化 AffectState,兼容旧版缺字段)
  1. dynamics.apply_time —— 激素/arousal衰减,好感度离线缓降,
     跨会话时重置耐心/warm_streak/回路计龄
  2. l3_store.working_memory 取最近对话 → extractor.extract(flash) ——
     离散分类他这条消息,含 persona_attack 标记
  3. dynamics.apply_events + transition —— 纯代码把分类结果变成数值变化,
     可能触发激素释放、好感度跨档(→ 埋下演化触发点)
  3.5 narrative.refresh —— 自我解释口径按需刷新(零LLM;生活正史事件做
     表面归因素材,经 life_sim.recent_event_texts 取)
  3.6 承诺到点催挂 ping(v0.6)—— 新承诺有明确到期 → 写 kind=commitment
     的 TimerPing(due+30min,触发前查回路防空催)
  4. l3_store.write_memory 落他的消息 + tags.assign_tags 打标(零LLM)
  5. 组装 L1:
       retrieval.retrieve(向量检索,排除工作记忆) +
       l3_store.cherished_memories + l2_hot.hot_memories →
       l1_assembly.build_memory_block(全局去重+弹性预算)
       identity.build_core_identity(读 chat.core_identity 覆盖)
       schedule.render_block(读 ScheduleItem,懒种默认作息)
       life_sim.pick_seed(读 LifeEvent,受 dull_streak/warm概率门控)
       guardrail.render_boundary_block(受 persona_attack 影响是否追加点破)
       notebook.render_block(读 Note)
       检索置信度提示(hits[0].score 过低 → memory_hint)
     全部喂给 injector.render 编译成一份 system_prompt
  6. 生成:
       若 tools_enabled → _generate_with_tools:有界 agent loop
         (client.chat_tools ⇄ tools.dispatch,最多 tool_max_rounds 轮,
          之后 tool_choice=none 强制作答;set_timer/write_note 直接落库)
       否则 → client.chat 单次生成
     _extract_timer 剥 <timer> 标签兜底(工具未开或模型偷懒时)
     _parse_generation 解析 <thinking>/<reply>(容错未闭合标签)
     guardrail.detect_break 检测崩坏 → 命中则 _apply_guardrail 一次纠正重生成
  7. _persist_agent_replies —— 她的每条回复独立写入 L3 + 打标
     (脑内剧场/情绪/刻骨铭心只挂第一条,避免情绪轴重复嵌入)
  8. timer_req(工具或标签产生的)→ 写 TimerPing;chat.affect 存回;
     timeline.record 落一行情绪快照(同一事务);commit
  9. 后台派发(不阻塞响应):
       _schedule_auto_dream()      —— 检查 pending 积压,够了后台跑 Dream
       life_sim.schedule_auto_sim  —— 检查新鲜事件池,不够则后台补写
       evolution.schedule_evolution —— 若本轮好感度升档触发了里程碑,后台演化

2.2 定时器闭环(跨请求、后台驱动)
--------------------------------------------------------------------------------
  她在步骤6生成时用 set_timer/<timer> 约定 → TimerPing(pending) 落库
  → TimerService 每5s轮询 → 到点先看 schedule.is_sleeping,睡着则顺延
    wake_ms + 随机抖动,否则认领(firing)
  → pipeline.handle_timer_fire:时间效应 → L1组装(含日程/话题种子/小本子/底线)
    → 生成(过守护层同款检测)→ 落L3 → ping标记fired
  → event_bus.publish → SSE 推给前端(掉线不影响,消息已在L3)

2.3 后台维护链路(离线,信号驱动,互不阻塞对话)
--------------------------------------------------------------------------------
  auto-dream:      pending记忆积压 → dream.run_dream(聚类/命名/合并/拆分/重映射)
  life_sim:        新鲜事件池不足 且 距上次生成够久
                    → life_sim.run_life_sim(LLM补写事件+计划,写入L3+ScheduleItem)
  night_agent:      作息判定睡觉 + 用户静默够久 + 每晚一次
                    → night_agent.run_night:
                        蒸馏(写L3 passage)+ 日记(写Note)+ 排明日(写ScheduleItem)
                        + notebook.housekeeping + health.compute_health(体检)
                        + dream.run_dream(积压够 或 健康亮旗时强制)
  persona演化:      好感度跨档触发里程碑(dynamics里判定,pipeline读取events)
                    → evolution.run_evolution:
                        config_store.snapshot → LLM提案 → append-only校验 → 应用

2.4 跨进程安全(v0.4)
--------------------------------------------------------------------------------
  四处后台任务(auto-dream/life_sim/night_agent/evolution)保留进程内
  asyncio.Lock 作第一道闸,真正的单飞由 db_locks.advisory_guard 保证
  (多worker下谁先拿到谁跑,拿不到让过)。定时器认领用 FOR UPDATE SKIP LOCKED,
  同一到点闹钟只被一个进程触发。heat flush 本就是相对增量UPDATE,天然安全。
  已知余留:SSE EventBus 是进程内的,多worker下主动消息可能推错进程——
  消息本体在L3,刷新即见,需要实时跨进程推送时再上 Redis pub/sub。

2.5 守护层的两个介入点
--------------------------------------------------------------------------------
  输入侧:extractor 每轮都判定 persona_attack(零额外调用,同一次flash里带出)
         → injector 在【底线】块里当轮追加点破
  输出侧:每轮生成后 guardrail.detect_break 扫描 replies
         → 命中则整轮多花一次pro调用重生成,重试仍崩也照发(绝不失声/绝不告警腔)

2.6 数据一致性关键约束
--------------------------------------------------------------------------------
  - 内容只有一份:L3.Memory.content 是唯一真相源。L2(热度)只持id;
    LifeEvent 只持 memory_id + 调度元数据;Note 是独立文档存储,不是Memory的副本。
  - 向量纯净:content_vec/emotion_vec 从不拼接 tag/时间/说话人,
    这些一律走旁挂列 + WHERE 过滤(tags GIN索引、ts_ms B-tree索引)。
  - LLM 权限边界:extractor/dynamics 组合保证"数值只由代码规则表决定";
    Dream/夜间代理/演化 是仅有的允许LLM写入结构化数据的三处离线场景,
    且各自都有明确的校验层(_validate / validate_payload / validate_proposal)
    和降级路径(失败不影响主流程)。
  - 配置变更留痕:chat.persona/core_identity/goal 的每次变更(无论user还是model)
    都先经 config_store.snapshot,append-only,可回退。
