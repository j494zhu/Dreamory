Part 1. 项目综合描述
现在的目标是构建一个伴侣app. 核心卖点是基于参数的情绪感知, 以及层级记忆库. 
层级记忆库非常类似于 Memgpt(Letta)
构建一个三级分层记忆系统,让对话 Agent 在有限上下文窗口下实现近乎无限、可持续的长期记忆。
每次处理一段对话,系统会产生三类原始数据:
文本消息本体(用户说的 / Agent 说的)
Agent 生成该消息时的 reasoning(思考链)
Agent 生成该消息时的情绪状态
记忆按访问速度/容量分三层:
L1 = 上下文窗口(快、小、当前在用)
L2 = Hot Zone 热缓冲(中、只读、常用记忆)
L3 = 全量冷存储(慢、无限、唯一真相源)
外加一个横跨 L3 的 Tag 注册表(不属于任何一层,是正交的索引结构)。
项目的后端使用fastapi, 前端使用html/css
项目的模型使用deepseek v4 pro以及deepseek v4 flash, embedding 模型使用 bge-m3, 关系表使用postgres
向量只编码纯内容:进入任何向量库的、被 embedding 的文本,只能是内容本身。tags、时间戳、说话人等元数据一律不拼进被嵌入的文本——它们会平移/污染向量,破坏余弦相似度的区分度。
这些元数据作为旁挂列,在检索时当 WHERE 过滤条件用。
LLM 只做"命名",不做"开放式打标签":热路径打 tag 全程零 LLM(确定性的向量匹配)。LLM 只在离线 Dream 阶段被调用,且只负责"给一个已经成形的聚类簇起名字"这一件它擅长的事。
派生数据可缓存,但必须可丢弃且单向派生:Hot Zone、物化视图等都是从 L3 派生的只读缓存,丢了能重建,绝不反向成为真相


Part 2. 项目结构描述
2.1. 情感参数集
我过去使用llm的时候观察到了一些痛点. llm本身是无状态的, 对于情绪没有直接的感知能力, 只能通过上下文去猜测. 故而尝试设计一个基于llm的伴侣模型的时候, 会出现很多不真实感. 
例如llm一直都是状态非常饱满的, 好像用户就算做了一件错误的事情, 说一句好话, llm马上就可以完全恢复状态. 
以及如果记录和llm进行的伴侣对话实验, 发现llm似乎每一条回复的长度都基本上一致. 
诸如此类还有很多. 我的结论是, 直接通过llm对话, 或者设计system prompt, 无法让llm在情感上表现的和人类类似. 
我认为人在对话交流的时候, 大脑内其实维护了一个类似状态机的隐藏参数集, 而这个参数集会影响实际的对话. 例如, 兴趣程度, 疲劳度, 唤起度等等. 
基于以上想法, 我现在准备在项目中采用以 Emotion Bids 和 Attachment Theory 理论为核心的情绪状态机来完成情绪模拟. 
部分代码派那段已经放在 /files 文件夹里了. 请使用这部分代码, 允许修改, 改编, 来构建情感模块
代码的核心思路为通过persona以及维护参数集的机制, 将参数代表的含义注入进prompt中. 
代码中会利用到llm的reasoning功能, 我称之为脑内剧场. 来强化实际对话中的情感真实和准确程度. 

2.2. 记忆库
记忆库的设计基于这一条理论: 显存是昂贵的, 内存是便宜的, 存储是免费的. 
所以我们可以将大量数据保存在硬盘中, 但是对于上下文需要进行精打细选. 
接下来描述数据库的结构. 
数据库分成三层, L1/L2/L3. 接下来分别讲解每一层的结构. 
L1 为非常重要的内容, 每次都会直接加载进对话. 
L1 包含以下板块: [核心人格, 记忆-刻骨铭心, 记忆-工作记忆, 记忆-L3检索, 当前目标, 当前参数情绪, 挂起回路]
核心人格指的是给llm设定的固有认知. 代表一个人思想中非常顽固的部分. 例如对于自我的元认知, 性别名字, 生日, 长期习惯等等. 这其中内容不是说完全一成不变, 但是需要保持一个非常强大的, 不容易被改变的惯性. 
有些内容需要特殊机制来改变, 例如年龄; 有些内容可以改变, 但是需要非常强大的理由. 例如爱好; 需要在对话中有足够的修改动力. 
在当前阶段, 先不需要考虑核心人格的改变, 暂且将其固化. 
记忆-刻骨铭心代表一些非常重要的记忆, 例如特别幸福的, 特别痛苦的, 这类造成了强烈的情绪波动的内容. 
在当前阶段, 假设这部分内容是不需要删除的. 
记忆-显存工作记忆代表最近的若干条对话记录, 也就是传统的messages数组的最后几条. 这里可以做一个预设, 例如k=10. 采用FIFO策略. 
记忆-L3检索代表从L3数据库中搜索出来的和当前对话相关的内容. 
当前情绪参数, OpenLoop等对应情感参数集的部分. 将其一起保存至L1

L2为热点层, 代表L3中高频使用出现的记忆. L2只保存id, 不保存文本消息, 这是为了保证数据一致性. id是文本在L3中的id
进入条件:时间衰减热度(time-decayed heat)——每次命中加分,分数随时间指数衰减。
不用裸 frequency,否则"陈年顶流"会赖着不走、挤占缓冲。
检索命中时,只在内存计数器里累加(hit_buffer[id] += 1、last_used_buffer[id] = now),不碰数据库。
后台攒批 flush:每隔 N 秒或攒够 M 条,一次批量 UPDATE 回主表。
容忍几秒延迟(热度是统计量,不是账本)。

淘汰策略:固定容量上限,满则按热度分数淘汰。直接采用成熟缓存算法:W-TinyLFU 或 LRU-K,不自造。
L3为原始数据层, 冷储存, 唯一数据源. 
主表(关系型) + VectorDB_1(内容) + VectorDB_2(情绪)
sql关系表的字段有: 
id: primary_key=uuid7, 身份 + 粗时间排序(字典序≈时间序), 
timestamp 整数/datetime,建 B-tree 索引, 精确时间查询用。区间查询、邻近查询都走此列,不解析 UUID
content, text文本消息本体,全库唯一一份
speaker enumuser / agent
emotion_reasoning text/json情绪 + reasoning 原文
tags array/json,旁挂元数据,检索时做过滤;不进向量
last_used 整数/datetime热度字段,后台批量回写
use_count整数热度字段,后台批量回写
必备索引:timestamp(区间/邻近查询)、use_count 或热度分数(取热记忆)、tags 的 GIN/倒排(标签过滤)。
向量数据库分别保存: 
VectorDB_1: (内容向量, id) —— 内容语义检索轴
VectorDB_2: (情绪/reasoning 向量, id) —— 情绪/心境检索轴

除了 L1/L2/L3, 数据库还包含 Tags 注册表(横跨 L3 的正交索引)
内容:受控词表(所有规范 tag)+ 每个 tag 的 centroid 原型向量(= 该 tag 名下所有成员记忆向量的质心)。
角色: 被编译进 L1 上下文(让 Agent 知道当前存在哪些 tag → 倾向复用、抑制发散);
被热路径用于打标签匹配;
注意:它不是一档"内存层",不进 L1/L2/L3 容量层级,是正交元数据结构。
被 Dream机制 维护(新建/合并/拆分/重映射)。
(尚不完全确定) Tag 组织建议:
按 facet(维度)而非自由标签。 预设若干正交维度(如 {领域, 实体, 类型, 项目, 时间桶}),
每条记忆在每个相关维度上取值。维度本身 3~6 个,天然框住 tag 数量,也让每个 facet 成为受控小词表(自动消解 replicant/replicants 这类问题)。
打 Tag(热路径,零 LLM)

原则:为检索打标签,不为描述打标签。 唯一职责是检索时把候选集切小;内容本身 + 向量已经"描述"了记忆。判断要不要打某 tag,只问:将来会不会用它来过滤? 不会就不打。

算法:kNN 标签传播 + 投票阈值(或 centroid 匹配)

给新消息打 tag:
  1. 用其内容向量在 VectorDB_1 找 top-k 邻居(或直接对各 tag 的 centroid 算相似度)
  2. 统计邻居携带的 tag,按相似度加权投票
  3. 仅当某 tag 的加权票数 ≥ 阈值 → 打上(过滤掉单个邻居的偶发噪声 tag)
  4. 若没有任何 tag 达到阈值(落在向量空间稀疏/未占领区)→ 标记为 pending,不打 tag
近义词、非字面 tag 自动解决:由 embedding 兜底("android" 向量近 "replicant";描述背叛但没出现"背叛"二字,向量仍近 betrayal)。这些是"意义轴"问题,归 embedding,不归 tag。
颗粒度/数量准则:
每条 2~5 个 tag 是甜区;>7 个基本是在用 tag 干 embedding 的活。
判别力检验:某 tag 出现在 >50% 记忆上 → 无区分度,删或拆维度;只出现在 1 条上 → 太细,是噪声(那条靠向量本就能搜到)。
打标签粒度:给蒸馏后的事实条目/passage 打 tag,不给原始消息流逐条打(贵且必然发散)。原始流靠纯文本/向量检索。
新 Tag 创建与整理(Dream / 离线)
热路径永远不创建新 tag,只分配已有 tag。新 tag 与词表维护全在离线 Dream 阶段。这把 LLM 关进"只命名"的小笼子,绕开"LLM 开放打标签不靠谱"的问题。
触发条件(信号驱动,不绑死"用户睡觉"):
tag 总数超阈值 / 词表熵升高 / 检测到空闲 / pending 堆积超阈值。
"睡觉/空闲"只是其中一种空闲信号。

Dream流程为记忆库维护. 
1. 聚类:把 pending 记忆 + 可疑区域的记忆做向量聚类
2. 命名:当某簇足够大且紧致 → 交 LLM 给该簇命名(LLM 唯一职责)
3. 合并:相似 tag 合并(如 androids → replicant)
4. 拆分:某 tag 成员在向量空间明显分成多坨 → 拆分(如 work → work-项目A / work-项目B)
5. 重映射:改写所有受影响记忆的 tag,并保留一张【旧tag → 规范tag】映射表
          (让老查询/老过滤器仍可翻译)
6. 更新:刷新受控词表 + 各 tag 的 centroid 原型向量
Dream 的设计约束:
黏性(stickiness):强烈偏向保留现有规范 tag,只在信号很强时才合并/拆分,避免 tag 身份每晚横跳(ml ↔ machine-learning)导致下游崩溃。
既合并也拆分:重复→合并,过度合并/碎片化→拆分。判据是"成员是否形成清晰子簇"。
重映射是主要工作量:不只改词表,要改写存量记忆并维护旧→新映射表。
Dream在对话中一般不进行, 仅仅在用户睡觉以后后台运行. 
Dream功能先做好, 但是暂时不开启. 

2.3 检索
检索流程:
  query
   → VectorDB_1(按内容)和/或 VectorDB_2(按情绪)语义召回 → 返回一批 id
   → 回主表取全字段(建议为 pgvector 内的 JOIN)
   → tags 作为 WHERE 过滤(向量=模糊意义轴,tag=硬类别轴,二者互补)
   → 命中记录热度(写内存计数器,见 L2)
   → 去重后填充 L1 对应记忆槽
双轴检索:可独立"按内容搜"或"按情绪搜"(如"找我感到被背叛的记忆"走 VectorDB_2),互不污染。
多步检索(可选增强):仿 MemGPT 的 function chaining,允许 Agent "搜一次→不满意→换词再搜"迭代。
条件偏置:用 L1 的 当前目标 给召回加权。

2.4 热度与缓存
热度 = 时间衰减分数(命中 +w,随时间指数衰减),而非裸 frequency。
命中只写内存计数器;后台攒批回写 L3。
L2 进出由热度分数 + 缓存淘汰算法(W-TinyLFU / LRU-K)决定。

2.5 其他策略
问题:同一条记忆可能既在 L2(因热)、又被当次检索从 L3 捞回,在 L1 重复占位。
推荐默认:组装 L1 时按 id 全局去重;优先级 刻骨铭心 > L2热 > 当次检索,同一 id 只保留最高优先级的那次,其余排除。
问题:刻骨铭心 / 工作记忆 / L3检索来的 三槽各占多少 token,溢出谁先被挤(MemGPT 队列管理问题)。
推荐默认(占 L1 记忆区总预算的比例,可调):

刻骨铭心:20%(几乎不驱逐,FIFO 仅在极端情况)
工作记忆/近期:50%(FIFO,最旧先出,出场前可触发摘要 summarization)
L3 检索来的:30%(按相关度排序,低分先丢)
溢出时:先压缩"工作记忆"为摘要,再驱逐"L3检索来的"低分项,最后才动"刻骨铭心"。

统一关系库 + 向量库为 PostgreSQL + pgvector:

直接把"回主表取数"变成真正的 SQL JOIN,而非跨存储 id 往返;
单一事务、单一真相源,天然契合铁律 1;
pgvector 支持 HNSW 索引,亚秒级近似检索。
(本地/笔记本场景若想更轻,可用 SQLite + 向量扩展;但要自己处理跨表对齐,不如 pgvector 省心。)

UUIDv7:用现成库生成,不要手搓大整数时间戳。
缓存层:用现成 W-TinyLFU / LRU-K 实现,不自造。
embedding 模型:内容轴与情绪轴各自固定一个模型,创建后不要中途更换(否则旧向量与新向量不可比)。


3. 实现建议:
建议实现优先级(里程碑)

M1 — L3 骨架:主表(UUIDv7 + timestamp 索引)+ VectorDB_1,实现写入与基础语义检索(回 id 取数)。验证铁律 1/2/3。
M2 — Tag 系统:Tag 注册表 + 热路径 kNN 打标签 + tag 过滤检索。先用手动种子词表冷启动。
M3 — 情绪轴:VectorDB_2 + 双轴检索。
M4 — L2 Hot Zone:热度分数 + 缓存淘汰 + 内存计数 + 批量回写。
M5 — L1 组装:各槽填充 + 决策 A/B 的去重与预算。
M6 — Dream:离线聚类 + LLM 命名 + 合并/拆分/重映射。
M7 — 增强:多步检索、当前目标条件偏置、OpenLoop。
